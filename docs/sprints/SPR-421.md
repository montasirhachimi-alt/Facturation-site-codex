# SPR-421 — Unified Search Provider Implementation

## Objective

SPR-421 replaces the placeholder CRM and Sales Unified Search providers from SPR-420 with real deterministic business search implementations.

This sprint does not add UI, migrate the Command Center, add Prisma queries, create APIs, add AI, implement fuzzy search or change navigation.

## Implemented Provider Behavior

Unified Search now returns real module-owned results through:

```text
Business modules
  ↓
SearchProvider
  ↓
SearchRegistry
  ↓
BusinessSearchRuntime
  ↓
SearchService
```

The Runtime remains module-agnostic. Concrete providers live inside business modules.

## Searchable Entities

CRM:

- Companies through `crmCompanyLocalService`
- Contacts through `crmContactLocalService`

Sales:

- Quotes through `quoteService`
- Invoices through `invoiceService`
- Sales Orders through `salesOrderService`
- Delivery Notes through `deliveryNoteService`
- Payments through `paymentService`

Payments are included because the current model exposes stable document number, invoice number, customer, status, method, reference and route data.

## Ranking Policy

Ranking is deterministic and explainable.

Field weights:

1. identifier/document number
2. primary title/name
3. secondary context fields
4. metadata/status fields

Match types:

1. exact normalized match
2. prefix match
3. substring match

The global Runtime then sorts by:

1. score descending
2. module ID
3. entity type
4. French title comparison
5. stable result ID

No fuzzy, semantic, vector, AI or random ranking is used.

## Route Policy

Every returned result uses an existing canonical route:

- `/crm/companies/<companyId>`
- `/crm/contacts/<contactId>`
- `/sales/quotes/<quoteId>`
- `/sales/invoices/<invoiceId>`
- `/sales/orders/<orderId>`
- `/sales/delivery-notes/<deliveryNoteId>`
- `/sales/payments/<paymentId>`

No route was created or invented.

## Workspace Isolation

Providers query existing workspace-scoped services with their canonical workspace IDs:

- CRM Companies and Contacts use `workspace-hicopilot`.
- Quotes, Invoices and Payments use the Sales quote workspace.
- Sales Orders use `sales-orders-main`.
- Delivery Notes use `sales-delivery-notes-main`.

Providers do not accept workspace identifiers from search text and do not return records from other workspaces.

## Module Filtering

Runtime module filtering continues to work.

Supported behavior:

- exact module filters such as `crm.contacts`
- group-style filters such as `crm` and `sales`
- unknown filters return no results

Filtering remains in the Runtime, not React.

## Provider Registration

Provider registration remains explicit through:

```text
src/services/search/search-provider.bootstrap.ts
```

Providers register once, outside React rendering, and duplicate provider registration remains rejected.

## Legacy Compatibility

`SearchService.search("dashboard")` continues to use the existing synchronous legacy module search path.

Structured calls such as `SearchService.search({ text: "Sara" })` and `SearchService.searchUnified(query)` use the Unified Search Runtime.

The current Command Center UI remains unchanged.

## Known Limitations

- Command Center still uses its existing local registries.
- No search UI consumes Unified Search yet.
- No Prisma/database search optimization was added.
- No permission-aware or user-specific search context exists yet.
- Meetings, Tasks and Notes remain prepared providers but are not returning results in this sprint.

## Explicit Non-Goals

No UI, fuzzy search, typo correction, semantic search, vector search, AI ranking, API endpoint, Prisma schema change, migration, route redesign, navigation redesign, timeline change or dashboard change was introduced.
