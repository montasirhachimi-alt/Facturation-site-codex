# SPR-419 — Business Timeline UX Hardening & Manual QA

## Summary

SPR-419 hardens the first production Business Timeline integration on Sales Order details.

The sprint keeps the existing SPR-418 architecture:

```text
Sales Order details
  ↓
SalesOrderBusinessTimeline
  ↓
TimelineService.getTimeline({ entityType: "sales.order", entityId })
  ↓
BusinessTimelineRuntime
  ↓
SalesTimelineProvider + InventoryTimelineProvider
  ↓
Shared Timeline UI
```

No Timeline provider, event type, Delivery Note behavior, reservation behavior, stock posting, quantity precision, Prisma schema, API or business workflow was changed.

## Clarified Acceptance Criteria

SPR-419A clarifies the acceptance criteria for SPR-419. This clarification does not change product behavior, Timeline architecture, providers, Runtime, `TimelineService`, Sales Orders, Inventory, Delivery Notes, reservations, stock logic, Prisma, APIs or routes.

### Conditional Authenticated Browser QA

Authenticated browser QA is required when the local environment exposes a valid authenticated Sales Order workflow.

If the current Runtime profile, configuration, feature flags or available demo credentials do not expose Sales Orders, browser QA must not fail SPR-419. In that case the sprint is accepted when the limitation is documented and all remaining architecture, runtime, rendering, accessibility and automated validation requirements pass.

Current environment-specific limitation:

- `alpha.crm-sales` remains the default Runtime profile.
- Sales Orders are inactive in `alpha.crm-sales`.
- `/sales/orders` redirects to `/dashboard` under the default profile.
- The internal `sales-operations` profile contains the needed modules, but is not runtime-selectable through the current local app configuration.
- The demo password required for browser login is not documented in the repository.

Policy:

```text
Authenticated browser QA is required when environment supports it.
Otherwise document the limitation and validate the remaining requirements.
```

### Graceful Degradation

If Timeline loading fails:

- the Sales Order page must remain fully usable;
- existing Sales Order functionality must continue working;
- the failure must remain local to the Timeline section;
- no global error boundary should appear;
- business data must remain accessible;
- retry must remain local to the Timeline.

This preserves the Timeline as a read-only contextual enhancement rather than a dependency for viewing or operating the Sales Order.

### Performance Invariants

Normal navigation between Sales Orders must not introduce unnecessary duplicate Timeline work.

The implementation should avoid:

- duplicate Timeline loads caused by unstable React effects;
- duplicate provider bootstrap;
- duplicate Timeline provider registration;
- duplicate event rendering;
- polling, WebSockets or server push for this local read-only surface.

This is an architectural-correctness invariant, not a micro-optimization requirement.

### Timeline Link Policy

Only documents with existing production routes should render as interactive Timeline links.

If no production route exists:

- display the event normally;
- render it as non-clickable;
- do not invent routes;
- do not create placeholder pages;
- do not expose unfinished navigation.

This prevents future Timeline integrations from fabricating navigation merely to make every event clickable.

## Initial QA Findings

Discovery found these concrete issues or constraints:

- Sales Order routes are activation-gated and redirect to `/dashboard` under the default `alpha.crm-sales` profile.
- The internal `sales-operations` profile contains `sales.orders`, `sales.delivery-notes`, `sales.products` and `inventory.stock`, but the runtime default remains Alpha and there is no environment switch wired into the current app.
- Browser login requires a demo password that is not documented in the repository. Direct browser session injection was refused by browser security policy, so authenticated browser QA could not reach `/sales/orders/[orderId]` in this session.
- The shared Timeline UI rendered events in a generic `div` rather than a semantic chronological list.
- Timeline status was primarily a color dot, which did not satisfy the hardening requirement that status not rely only on color.
- The integration already cleared events on load and used cleanup cancellation, but late-response rejection was implicit rather than explicit.
- Error state was local but had no retry action.

## UX Corrections

### Timeline Event Hierarchy

`TimelineEvent` now presents each event with a clearer generic hierarchy:

1. title
2. date and actor
3. visible status badge
4. description
5. optional link

No raw metadata, provider IDs or internal entity IDs are displayed.

### Status Visibility

Status is no longer color-only. Each event shows a compact textual badge:

- `Info`
- `Suivi`
- `Validé`
- `Attention`
- `Alerte`

The colored dot remains decorative support only.

### Responsive Wrapping

Event titles, descriptions, actor labels and link labels now use safer wrapping/truncation so long document numbers or descriptions do not force horizontal overflow.

### Semantic Structure

`TimelineCard` now renders events as:

```text
ol
  li
    article
```

This keeps the shared Timeline UI generic while improving accessibility for chronological content.

### Error Retry

`SalesOrderBusinessTimeline` now exposes a local `Réessayer` action in the error state.

The retry only increments local presentation state and calls the same `TimelineService` read path. It does not refresh the page, mutate business records or register providers again.

### Stale Result Protection

The integration now uses an explicit request id guard:

```text
requestIdRef
```

Late responses from older Sales Order loads cannot replace the latest requested timeline. Previous events are cleared before each load.

## Navigation Validation

- Payments: `/sales/payments/[paymentId]`

- Inventory workspace: `/inventory`

Route availability note:

- `/inventory`, `/sales/orders` and `/sales/delivery-notes` are inactive in default `alpha.crm-sales`.
- They are intended for controlled profiles such as `sales-operations`.
- Under default Alpha, `/sales/orders` redirects to `/dashboard`.

No fake route or placeholder link was added.

## Refresh Behavior

Refresh remains narrow and local.

The Sales Order details page passes a `timelineRefreshKey` based on:

- Sales Order `updatedAt`
- linked Delivery Note `updatedAt` / `postedAt`
- Inventory movement count

When that key changes, the Timeline reloads through `TimelineService`.

No polling, WebSocket, global state library, backend endpoint or synthetic Timeline event was introduced.

## Responsive QA

- Timeline card remains constrained inside the existing Sales Order page column
