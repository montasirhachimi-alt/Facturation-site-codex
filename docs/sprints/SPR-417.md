# SPR-417 — Inventory & Delivery Timeline Provider

## Summary

SPR-417 adds the logistics contribution to the Business Timeline Engine.

The new `InventoryTimelineProvider` translates existing Inventory reservation movements, Delivery Notes and physical Inventory `ISSUE` movements into generic `TimelineEvent` records. It connects physical execution to the commercial journey without changing Sales Orders, Delivery Notes, Inventory posting, reservations, Prisma, APIs or UI pages.

## Existing Canonical Sources

The provider reads current module-owned sources:

- Reservations and releases: `StockMovement` records with `type: "RESERVATION"` or `type: "RELEASE"`
- Physical stock issues: `StockMovement` records with `type: "ISSUE"`
- Inventory source: `inventoryLocalService.getSnapshot().movements`
- Delivery Notes: `deliveryNoteService`
- Sales Orders: `salesOrderService`

Canonical relationship fields used:

- `StockMovement.referenceType`
- `StockMovement.referenceId`
- `DeliveryNote.salesOrderId`
- `DeliveryNoteLine.salesOrderLineId`
- `DeliveryNote.id`
- `SalesOrder.id`

Canonical timestamps used:

- `StockMovement.postedAt`
- `StockMovement.createdAt`
- `DeliveryNote.createdAt`
- `DeliveryNote.updatedAt`
- `DeliveryNote.postedAt`
- `DeliveryNote.archivedAt`

## Provider Design

One provider was added:

```text
InventoryTimelineProvider
```

This single provider owns logistics events because reservation, physical issue and delivery-note posting are one execution chain in the current architecture.

Location:

```text
src/modules/inventory/timeline/
  inventory-timeline.data-source.ts
  inventory-timeline.mapper.ts
  inventory-timeline.provider.ts
  inventory-timeline.types.ts
  inventory-timeline.utils.ts
  index.ts
```

Delivery Note reconstruction is included through a narrow read-only data-source interface rather than by changing Delivery Note behavior.

## Supported Entities

Supported timeline query entity types:

- `sales.order`
- `delivery.note`
- `inventory.movement`
- `inventory.reservation`

The provider intentionally does not claim `sales.quote`, `sales.invoice` or `sales.payment`; those remain owned by `SalesTimelineProvider`.

## Supported Events

Generated event types:

- `inventory.reservation.created`
- `inventory.reservation.released`
- `delivery.note.created`

- `delivery.note.posted`

Events are generated only when the relevant canonical records and timestamps exist.

## Journey Resolution

Sales Order query:

```text
SalesOrder.id
  ↓
DeliveryNote.salesOrderId
  ↓
StockMovement.referenceType = DELIVERY_NOTE
  ↓
StockMovement.referenceId = DeliveryNote.id
```

Reservation query:

```text
StockMovement.type = RESERVATION / RELEASE
StockMovement.referenceType = SALES_ORDER
StockMovement.referenceId = SalesOrder.id
```

Delivery Note query:

```text
DeliveryNote.id
  ↓
DeliveryNote.salesOrderId
  ↓
StockMovement.referenceType = DELIVERY_NOTE
  ↓
StockMovement.referenceId = DeliveryNote.id
```

Inventory Movement query:

```text
StockMovement.id
  ↓
referenceType / referenceId
  ↓
Sales Order or Delivery Note only when explicitly linked
```

No relationship is inferred from product name, SKU, quantity, totals, dates, warehouse, customer or document-number similarity.

## Partial Delivery Handling

Each posted Delivery Note produces its own deterministic event.

For the validated logistics pattern:

```text
Reserve 8
Deliver 3
Deliver 5
```

The provider represents:

- the reservation movement as `inventory.reservation.created`
- the first posted BL as `delivery.note.posted`
- the first physical execution milestone as `delivery.note.partially_delivered`
- the first `ISSUE` movement as `inventory.issue.posted`
- the second posted BL as `delivery.note.posted`
- the final physical execution milestone as `sales.order.fully_delivered`
- the second `ISSUE` movement as `inventory.issue.posted`

The provider does not reconstruct partial delivery from final Sales Order aggregates alone. It requires separate posted Delivery Notes or explicit Inventory movements.

## Event Ownership

`SalesTimelineProvider` owns commercial document events:

- Quote
- Sales Order commercial creation/update
- Invoice
- Payment

`InventoryTimelineProvider` owns logistics execution events:

- reservation/release movements
- Delivery Note creation/posting
- partial/final physical delivery milestones
- Inventory `ISSUE` movements

This prevents duplicate `sales.order.updated` or generic status events.

## Determinism

Event IDs combine:

```text
inventory-timeline + rootType + rootId + sourceType + sourceId + eventType + canonicalTimestamp
```

Returned events are normalized by the Timeline Engine:

- filtered by requested entity
- deduplicated by event id
- sorted newest-first
- returned as immutable results

The provider is read-only and does not mutate Sales Orders, Delivery Notes, Inventory balances, reservations or stock movements.

## Architecture

Dependency direction:

```text
Inventory / Delivery canonical data
  ↓
InventoryTimelineProvider
  ↓
TimelineProvider contract
  ↓
BusinessTimelineRuntime
  ↓
TimelineService
```

Registration is explicit through the existing timeline bootstrap:

```text
src/services/timeline/timeline-provider.bootstrap.ts
```

The generic Timeline Runtime and shared Timeline UI do not import Inventory, Delivery Notes or Sales modules.

## Relationship Limitations

- Reservation consumption during Delivery Note posting updates reserved balances but does not create a separate persisted consumption movement. Therefore `inventory.reservation.partially_consumed` and `inventory.reservation.consumed` are not generated yet.
- Full delivery is emitted when a posted Delivery Note is the latest posted Delivery Note for a Sales Order whose current status is `delivered`.
- Delivery Note line-level timeline events are not generated separately unless required later.
- Delivery Notes are not connected back to Quotes unless Sales Order / Quote relationships are already represented by the Sales provider.
- Inventory balance history is not reconstructed because balances are current aggregates.
