## Summary

SPR-416 adds the first official Business Timeline provider: `SalesTimelineProvider`.

The provider translates existing Sales records into generic `TimelineEvent` objects consumed by the Runtime-first Business Timeline Engine introduced in SPR-415. It does not modify Sales workflows, Delivery Notes, Inventory posting, Prisma, APIs or existing UI pages.

## Existing Sales Sources

The provider reads the existing Sales module stores and services:

- Quotes: `quoteService` from `src/modules/sales/quotes/quote.store.ts`
- Sales Orders: `salesOrderService` from `src/modules/sales/orders/order.store.ts`
- Invoices: `invoiceService` from `src/modules/sales/invoices/invoice.store.ts`
- Payments: `paymentService` from `src/modules/sales/payments/payment.store.ts`

The provider uses the current canonical workspaces:

- Quotes, Invoices and Payments: `SALES_QUOTES_WORKSPACE_ID`
- Sales Orders: `SALES_ORDERS_WORKSPACE_ID`

No server repository, Prisma client, API route or persistence write is used by the provider.

## Supported Entities

The provider supports these timeline query entity types:

- `sales.quote`
- `sales.order`
- `sales.invoice`
- `sales.payment`

It also accepts compatibility aliases such as `Quote`, `SalesOrder`, `Invoice` and `Payment`.

## Journey Resolution

The provider resolves related documents only through explicit identifiers:

- Quote to Sales Order: `SalesOrder.sourceQuoteId`
- Quote to Invoice: `Invoice.quoteId`
- Invoice to Payment: `Payment.invoiceId`
- Payment to Invoice: `Payment.invoiceId`
- Invoice to Quote: `Invoice.quoteId`
- Sales Order to Quote: `SalesOrder.sourceQuoteId`

It does not connect documents by customer name, email, totals, document number similarity or approximate dates.

Because the generic Timeline Engine filters events by requested `entityType` and `entityId`, related Sales document events are attached to the requested root timeline while preserving the actual source document in serializable metadata:

- `sourceEntityType`
- `sourceEntityId`
- `sourceDocumentNumber`
- `sourceDocumentType`

## Determinism

Event IDs are stable and combine:

```text
sales-timeline + rootType + rootId + sourceType + sourceId + eventType + canonicalTimestamp
```

Dates come from existing Sales fields:

- `createdAt`
- `updatedAt`
- `issueDate`
- `orderDate`
- `receivedAt`
- `archivedAt`

The Timeline Engine still handles deduplication, sorting newest-first and immutable results.

## Relationship Limitations

Current Sales records do not store a full status transition history.

Consequences:

- Intermediate status history cannot be reconstructed.
- Events such as exact Quote acceptance time use the currently persisted `updatedAt` transition timestamp.
- Invoice paid/partially paid history is primarily represented by linked Payment events.
- Sales Order to Invoice relationship is only available through a shared source Quote when that relationship exists.
- Delivery Notes are intentionally not included in this Sales provider sprint.
