# SPR-418 — Business Timeline UI Integration

## Summary

SPR-418 integrates the Business Timeline into the first production workflow: Sales Order details.

The Sales Order detail workspace now displays a read-only `Historique de l'activité` section after the order lines. The section uses the existing `TimelineService.getTimeline({ entityType: "sales.order", entityId })` path and renders generic `TimelineEvent` records through the shared Timeline UI.

No Sales, Inventory, Delivery Note, reservation, posting, quantity, Prisma, API or business workflow behavior was changed.

## Integration Target

Integrated surface:

```text
Sales Order details
/sales/orders/[id]
```

Placement:

```text
Header
Metrics
Order lines
Historique de l'activité
Edit dialog
```

This keeps the page story intact: commercial identity, order content, then cross-module journey.

## Loading Boundary

The selected Sales Order detail page is a client workspace. The Timeline path used here is client-safe in the current Alpha architecture:

- `SalesOrderBusinessTimeline` imports `TimelineService`.
- `TimelineService` bootstraps already registered Timeline providers through `src/services/timeline/timeline-provider.bootstrap.ts`.
- Providers read existing client-side module services/local stores.
- No Prisma, server repository, database adapter or API route is imported by the UI component.

The component owns only presentation state:

- `loading`
- `ready`
- `error`
- returned `TimelineEvent[]`

It clears previous events before each load so a changed Sales Order cannot display stale events from another order.

## Component Structure

```text
src/modules/sales/orders/ui/order-details-workspace.tsx
  Owns the existing Sales Order page and passes the selected order id.

src/modules/sales/orders/ui/sales-order-business-timeline.tsx
  Loads the timeline through TimelineService and handles local UI states.

src/ui/timeline/
  Renders generic TimelineEvent records without knowing Sales or Inventory.
```

## Timeline Service Usage

The integration calls:

```ts
timelineService.getTimeline({
  entityType: "sales.order",
  entityId: salesOrderId
});
```

No UI code queries Quote, Invoice, Payment, Delivery Note, reservation or Inventory movement sources directly to reconstruct history.

## Architecture

The UI integration follows the provider architecture introduced in SPR-415 through SPR-417:

```text
Sales Order Details
  ↓
SalesOrderBusinessTimeline
  ↓
TimelineService.getTimeline({ entityType: "sales.order", entityId })
  ↓
BusinessTimelineRuntime
  ↓
Registered providers
  ↓
Generic TimelineEvent[]
  ↓
BusinessTimeline UI
```

The Sales Order details page does not reconstruct events from Sales, Delivery Note or Inventory stores. It only passes the Sales Order id to the timeline component.

## UI States

The new section supports:

- Loading: compact skeleton rows inside the timeline section, announced with `role="status"` and `aria-busy`.
- Empty: `Aucune activité disponible pour cette commande.`
- Error: a local red error panel; the rest of the Sales Order page remains usable.
- Success: returned events are passed directly to `BusinessTimeline`.

French copy:

- Title: `Historique de l'activité`
- Empty state: `Aucune activité disponible pour cette commande.`
- Error state: `Impossible de charger l'historique de cette commande.`

## Event Presentation

The shared Timeline UI displays:

- title
- description
- date/time using the existing `Intl.DateTimeFormat("fr-FR")` pattern
- status indicator
- actor label when present
- event link when present

`TimelineEvent` now uses Next.js `Link` for event links so internal navigation preserves SPA behavior. Events without links remain non-interactive.

## Refresh Behavior

The Sales Order details page passes a stable refresh key based on:

- Sales Order `updatedAt`
- linked Delivery Note update/posting timestamps
- Inventory movement count

This lets the read-only timeline reload after local store updates without making the timeline component import module-specific stores.

## Read-Only Guarantee

The implementation does not call any mutation path:

- no Sales Order create/update/confirm/cancel path
- no Delivery Note create/post/archive path
- no Inventory posting path
- no reservation/release path
- no persistence write endpoint
- no Prisma model or migration

Rendering the Timeline only reads generic `TimelineEvent[]` from the Timeline Runtime.

## Import Safety

Confirmed boundaries:

- `src/ui/timeline/` remains domain-agnostic.
- Timeline UI does not import Sales, Inventory, Delivery Notes or CRM modules.
- Sales Order timeline UI imports `TimelineService`, not timeline providers.
- Provider registration remains in the service bootstrap, outside React rendering.
- Runtime validation now checks those boundaries.

## Known Limitations

- Only Sales Order details have a production Timeline placement.
- Quote, Invoice, Payment, Delivery Note, Company and Contact details do not render the Business Timeline yet.
- Some future event types remain unavailable because canonical timestamps/history do not exist yet.
- Reservation consumption events are still not emitted separately because the current posting model does not persist a dedicated consumption movement.
- The timeline refreshes on existing Sales Order, Delivery Note and Inventory store updates; no polling, WebSocket or global invalidation system was added.
