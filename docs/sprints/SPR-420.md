# SPR-420 — Unified Global Search Foundation

## Summary

SPR-420 introduces the Unified Global Search Foundation.

This sprint is infrastructure only. It does not add a search dialog, search input, keyboard shortcut, command palette, AI ranking, SQL search, Prisma queries, APIs or UI changes.

## Architecture

The new foundation follows the Runtime-first pattern already used by Business Timeline:

```text
Business Modules
  ↓
Search Providers
  ↓
Search Registry
  ↓
BusinessSearchRuntime
  ↓
SearchService
  ↓
Future UI / Command Center / HicoPilot / Agents
```

React consumers must call `SearchService`. They must not call the Search Runtime or business modules directly.

## Search Contracts

Runtime package:

```text
src/runtime/search/
  business-search.types.ts
  business-search.registry.ts
  business-search.service.ts
  business-search.utils.ts
  index.ts
```

Canonical query:

```ts
type SearchQuery = {
  text: string;
  limit?: number;
  modules?: readonly ModuleId[];
};
```

Canonical result:

```ts
type SearchResult = {
  id: string;
  entityType: string;
  entityId: string;
  moduleId: ModuleId;
  title: string;
  subtitle?: string;
  description?: string;
  keywords?: readonly string[];
  icon?: string;
  url?: string;
  score: number;
  metadata?: Readonly<Record<string, unknown>>;
};
```

Provider contract:

```ts
type SearchProvider = {
  moduleId: ModuleId;
  label?: string;
  search(query: SearchQuery, context: SearchProviderContext): SearchProviderResult;
};
```

## Registry

`SearchRegistry` owns provider registration.

Responsibilities:

- register providers by `moduleId`
- reject duplicate providers
- list providers deterministically
- unregister providers for tests and future profile changes
- keep provider metadata immutable

The registry has no React, Next.js, Prisma, server repository, UI or business-page dependency.

## Runtime

`BusinessSearchRuntime` owns aggregation.

Responsibilities:

- validate and normalize `SearchQuery`
- invoke registered providers
- filter providers by `query.modules`
- merge provider results
- deduplicate by result `id`
- sort deterministically
- apply `limit` after sorting
- isolate provider failures

Provider failures are captured in diagnostics and do not prevent healthy providers from returning results.

## SearchService

`SearchService` remains the public application facade.

Compatibility is preserved:

- `search("dashboard")` continues to use the existing synchronous module search.
- `search({ text: "atlas" })` uses the new Unified Search Runtime.
- `searchUnified(query)` and `searchUnifiedWithDiagnostics(query)` expose explicit future-safe calls.

This lets future UI migrate to the canonical Runtime search without breaking current Header Search or Command Center behavior.

## Initial Providers

SPR-420 registers placeholder providers only for current Alpha CRM/Sales workspaces:

- `crm.overview`
- `crm.companies`
- `crm.contacts`
- `crm.meetings`
- `crm.tasks`
- `crm.notes`
- `sales.quotes`
- `sales.invoices`
- `sales.payments`

They return empty arrays intentionally. Full CRM, Sales, Inventory or repository search is future work.

Provider registration happens through:

```text
src/services/search/search-provider.bootstrap.ts
```

Registration is outside React rendering.

## Determinism

Search results are sorted by:

1. score descending
2. module ID
3. entity type
4. French title comparison
5. result ID

This gives stable output for identical inputs.

## Failure Isolation

If one provider throws, `BusinessSearchRuntime.searchWithDiagnostics()` records a provider failure and continues aggregating other providers.

`BusinessSearchRuntime.search()` returns only the healthy normalized result list.

## Runtime Validation

Runtime validation now covers:

- provider registration
- duplicate prevention
- deterministic ordering
- module filtering
- limit behavior
- provider failure isolation
- `SearchService` facade usage
- legacy string search compatibility
- provider registration outside React
- Runtime import safety
- shared UI avoiding direct Runtime calls
- Platform Search avoiding direct module-provider imports

## Known Limitations

- No business repository search is implemented yet.
- Existing Command Center record search still uses its current local registry.
- Placeholder providers return empty results.
- No permissions, workspace or tenant filter is added to `SearchQuery` yet.
- No UI consumes the new Runtime search yet.

## Future Extensions
